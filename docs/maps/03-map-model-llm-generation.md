# Map Model — LLM-Generated, JSON-Loaded

> **Status:** Draft Design

## 1. Overview

Maps are authored as JSON files, generated by an LLM prompted with real-world geographical context. The pipeline is:

```
Real Geography (text / map data)
        │
        ▼
   LLM Prompt
        │
        ▼
  MapBlueprint.json   ← high-level description (zones, terrain, POIs, entities)
        │
        ▼
   Map Compiler        ← deterministic expansion into per-chunk data
        │
        ▼
  world.json + chunks/*.json + zones.json   ← runtime files loaded by the engine
```

The LLM never produces raw tile arrays. It produces a **MapBlueprint** — a compact, semantic description of the map — which a deterministic compiler expands into the full tile-level data the engine consumes.

## 3. Terrain Generation Rules

The LLM describes terrain as **regions** (high-level shapes). The map compiler expands them into per-tile data.

### 3.1 Region Types

| Region Type | LLM Provides | Compiler Generates |
|-------------|-------------|-------------------|
| `elevation` | Polygon + height range + profile | Per-tile `HeightLevel` via noise-seeded interpolation within polygon |
| `river` | Path (chunk waypoints) + width + depth | Carved channel tiles with `WaterLevel > HeightLevel`, bank tiles with shore surface |
| `crater` | Center + radius + rim/floor height | Circular bowl: heights descend from rim to floor, optional irradiated flag |
| `lake` | Center + radius + depth | Flat `WaterLevel` region, terrain carved beneath |
| `coastline` | Edge + depth gradient | Edge tiles slope to 0, water fills to given level, beach strip |
| `flat` | Area + height | Uniform height across area (default for uncovered tiles) |

### 3.2 Profile Types (for `elevation` regions)

| Profile | Meaning |
|---------|---------|
| `flat` | Uniform height (plateaus) |
| `rolling_hills` | Perlin noise between height range, gentle |
| `rugged` | Higher frequency noise, steeper slopes |
| `terraced` | Quantized height steps (rice paddy effect) |
| `cliff_edge` | Sharp transition on one side, gradual on the other | 

### 3.3 Surface Assignment

Surfaces are assigned per-region. Overlapping regions use priority order (last region wins). The `coverage` + `secondary` + `tertiary` fields distribute surface types using noise:

```
For each tile in region:
    noise = perlin(tile.x, tile.y, region.seed)
    if noise < coverage:
        surface = primary
    elif noise < coverage + secondaryCoverage:
        surface = secondary
    else:
        surface = tertiary
```

### 3.4 Road Carving

Roads are carved **after** terrain heights are set:
- Path is expanded to the specified width using perpendicular offset
- Road tiles get the specified `surfaceType` (typically `Asphalt`)
- Height is **smoothed** along the road path (roads don't climb cliffs vertically)
- `condition` controls how much rubble/breakup is mixed in (lower = more `Rubble` tiles replacing the road surface)

## 4. NPC Integration

NPCs are placed in the blueprint and carry references into the existing `EntitySpawnInfo`, `DialogueTree`, and `FactionDefinition` systems.

### 4.1 NPC Blueprint Schema

```jsonc
"npcs": [
  {
    "id": "npc_elder_mae",
    "name": "Elder Mae",
    "prefabId": "npc_elder",
    "placement": {
      "mode": "building",
      "buildingRef": "bld_elder_house",
      "room": "main_hall"
    },
    "faction": "settlers_alliance",
    "level": 8,
    "dialogueId": "dlg_elder_mae",
    "persistent": true,
    "schedule": {
      "dawn":  { "location": "bld_elder_house", "activity": "idle" },
      "day":   { "location": "bld_elder_house", "activity": "idle" },
      "dusk":  { "location": "bld_elder_house", "activity": "idle" },
      "night": { "location": "bld_elder_house", "activity": "sleep" }
    },
    "vendorInventory": null,
    "questGiver": ["q_rat_hunt", "q_safe_passage"],
    "tags": ["essential", "quest_giver"]
  },
  {
    "id": "npc_trader_vik",
    "name": "Trader Vik",
    "prefabId": "npc_trader",
    "placement": {
      "mode": "tile",
      "chunkX": 3,
      "chunkY": 5,
      "localTileX": 10,
      "localTileY": 10
    },
    "faction": "settlers_alliance",
    "level": 5,
    "dialogueId": "dlg_trader_vik",
    "persistent": true,
    "schedule": {
      "dawn":  { "location": "tile:3,5:10,10", "activity": "setup_shop" },
      "day":   { "location": "tile:3,5:10,10", "activity": "vending" },
      "dusk":  { "location": "bld_elder_house", "activity": "walk_to" },
      "night": { "location": "bld_elder_house", "activity": "sleep" }
    },
    "vendorInventory": "vendor_general_goods",
    "questGiver": [],
    "tags": ["vendor"]
  }
]
```

### 4.2 NPC Placement Modes

| Mode | Description | Compiler Action |
|------|-------------|-----------------|
| `building` | Inside a building (large buildings only) | Places `EntitySpawnInfo` in the building's interior chunk. Uses `room` field to determine position within interior layout. |
| `tile` | Specific tile coordinate | Places `EntitySpawnInfo` directly in the specified chunk at (localTileX, localTileY). |
| `zone` | Anywhere in a zone | Compiler picks a walkable, non-water, non-building tile within the zone. Seed-deterministic. |
| `road` | Along a road | Compiler picks a road tile from the specified `roadRef`. |
| `building_entrance` | Outside a building door | Compiler places at the entrance tile of the referenced building. |

### 4.3 Schedule System

The `schedule` field maps to the 4 day phases from `DayNightCycleProcessor` (Dawn, Day, Dusk, Night). Each entry specifies:

- **location**: A building ref (`bld_*`) or tile coordinate (`tile:cx,cy:lx,ly`)
- **activity**: What the NPC does at that location

Activities map to AI behavior states:

| Activity | AI Behavior |
|----------|-------------|
| `idle` | Stand/sit at location, ambient animations |
| `sleep` | Unresponsive until woken, inside building |
| `patrol` | Walk a loop between waypoints near location |
| `vending` | Stand at shop position, enable trade interaction |
| `setup_shop` | Walk to vending location (transition) |
| `walk_to` | Pathfind to specified location |
| `guard` | Stand at entry point, alert to hostiles |
| `work` | Ambient work animations (hammering, cooking) |

The `DayPhaseChangedEvent` triggers NPC schedule transitions. NPCs not at their scheduled location pathfind there over real time.

### 4.4 NPC-to-System Mapping

```
Blueprint NPC
  ├── prefabId         → selects visual model / animation set
  ├── faction          → FactionComponent (reputation system)
  ├── dialogueId       → DialogueTree (dialogue system)
  ├── level            → StatsComponent + HealthComponent scaling
  ├── vendorInventory  → InventoryComponent (trade system)
  ├── questGiver[]     → QuestProcessor (quest availability check)
  ├── persistent       → SaveService (survives chunk unload)
  └── tags             → gameplay flags ("essential" = can't die)
```

## 5. Enemy Integration

### 5.1 Enemy Group Blueprint Schema

Enemies are placed as **groups** rather than individuals. A group defines composition, territory, and respawn behavior.

```jsonc
"enemyGroups": [
  {
    "id": "eg_radrats_ruins",
    "name": "Ruins Radrat Pack",
    "faction": "wildlife",
    "zone": "zone_downtown_east",
    "territory": {
      "chunkX": 1,
      "chunkY": 1,
      "radiusTiles": 20
    },
    "composition": [
      { "prefabId": "enemy_radrat", "count": 3, "level": [1, 3] },
      { "prefabId": "enemy_radrat_alpha", "count": 1, "level": [3, 5] }
    ],
    "spawnRule": "scattered",
    "respawn": {
      "enabled": true,
      "cooldownHours": 24,
      "conditionFlag": null
    },
    "lootTable": "loot_radrat",
    "aggroRadius": 8,
    "patrolPath": null,
    "tags": ["wildlife", "hostile"]
  },
  {
    "id": "eg_raiders_camp",
    "name": "Burnside Raiders",
    "faction": "wasteland_raiders",
    "zone": "zone_downtown_east",
    "territory": {
      "chunkX": 2,
      "chunkY": 2,
      "radiusTiles": 16
    },
    "composition": [
      { "prefabId": "enemy_raider_melee", "count": 2, "level": [2, 4] },
      { "prefabId": "enemy_raider_ranged", "count": 2, "level": [2, 4] },
      { "prefabId": "enemy_raider_boss", "count": 1, "level": [5, 5], "nameOverride": "Scar", "dialogueId": "dlg_scar_taunt" }
    ],
    "spawnRule": "clustered",
    "respawn": {
      "enabled": false,
      "conditionFlag": "raiders_camp_cleared"
    },
    "lootTable": "loot_raider",
    "aggroRadius": 12,
    "patrolPath": [
      { "chunkX": 2, "chunkY": 2, "localTileX": 4, "localTileY": 4 },
      { "chunkX": 2, "chunkY": 2, "localTileX": 12, "localTileY": 4 },
      { "chunkX": 2, "chunkY": 2, "localTileX": 12, "localTileY": 12 }
    ],
    "tags": ["human", "hostile"]
  }
]
```

### 5.2 Spawn Rules

| Rule | Description |
|------|-------------|
| `scattered` | Compiler distributes enemies randomly within territory radius on walkable tiles. Minimum 3 tiles between each entity. |
| `clustered` | All enemies placed within 5 tiles of territory center. Camp-style grouping. |
| `perimeter` | Enemies placed along the edge of the territory radius. Guard-style. |
| `patrol` | Enemies assigned to `patrolPath` waypoints. Cycle through in order. |
| `ambush` | Enemies placed on high ground or behind cover near a road. Hidden until player is within `aggroRadius`. |

### 5.3 Respawn Behavior

- `enabled: true` — group respawns `cooldownHours` (in-game hours) after being fully killed
- `enabled: false` — group is permanently dead once cleared (one-time encounter)
- `conditionFlag` — if set, the group does NOT spawn when this `WorldStateService` flag is `true`. Used for quest-gated encounters (e.g., "once the camp is cleared, it stays cleared").

### 5.4 Enemy-to-System Mapping

```
Blueprint EnemyGroup
  ├── faction          → FactionComponent (hostile to player if rep < hostileThreshold)
  ├── composition[]
  │   ├── prefabId     → visual model + animation + AI template
  │   ├── level[]      → random level in range → scales stats via GAME_CONSTANTS formulas
  │   ├── nameOverride → custom display name (bosses)
  │   └── dialogueId   → bark/taunt dialogue (optional, played on aggro)
  ├── lootTable        → rolled per kill via existing loot system
  ├── aggroRadius      → AI sight range override (default 20 tiles from GAME_CONSTANTS)
  ├── patrolPath       → AI patrol waypoints
  └── tags             → "hostile" triggers combat, "wildlife" for animal AI
```

### 5.5 Difficulty Scaling

The `enemyDifficultyTier` on the zone combined with the enemy `level` range gives the map designer (or LLM) control over difficulty:

| Zone Tier | Typical Enemy Levels | Example Enemies |
|-----------|---------------------|-----------------|
| 1 | 1–3 | Radrats, stray dogs |
| 2 | 2–5 | Raiders, mutant insects |
| 3 | 4–8 | Armed raiders, feral ghouls |
| 4 | 7–12 | Irradiated beasts, raider elites |
| 5 | 10–15 | Boss mutants, power-armored enemies |

## 7. Quest Hook Integration

Quest definitions live in separate `quests/*.json` files (per existing schema). The map blueprint doesn't redefine quests — it **hooks** them into the map by declaring where quest-relevant entities, triggers, and objectives are placed.

### 7.1 Quest Hook Schema

        "description": "Elder Mae gives the quest when first spoken to"

        "stageId": "stage_kill_rats",

### 7.2 Quest Hook Types

**Triggers** — what starts the quest:

| Type | Ref Field | Description |
|------|-----------|-------------|
| `npc_dialogue` | `npcRef` | Quest starts from a dialogue choice with this NPC |
| `quest_complete` | `prerequisite` | Auto-triggers when another quest finishes |
| `zone_enter` | `zoneRef` | Triggers on entering a zone (via `ZoneEnteredEvent`) |
| `item_pickup` | `itemId` | Triggers when player picks up a specific item |
| `container_open` | `containerRef` | Triggers when player opens a specific container |

**Objectives** — what the player must do:

| Type | Key Fields | Maps To |
|------|-----------|---------|
| `kill_count` | `enemyGroupRef`, `targetTag`, `count` | `KillTracker` + `WorldStateService` counter |
| `kill_named` | `enemyGroupRef`, `targetName` | `EntityDiedEvent` with name match |
| `interact_container` | `containerRef` | Container interaction event |
| `interact_npc` | `npcRef` | Dialogue completion event |
| `reach_tile` | `chunkX/Y`, `localTileX/Y` | Player position check |
| `escort_npc` | `npcRef`, destination | NPC alive + at destination |
| `flag_set` | `flag` | `WorldStateService` flag check |

**Rewards** — what happens on completion (references, not definitions):

| Type | Description |
|------|-------------|
| `world_flag` | Sets a WorldStateService flag (gates spawns, containers, etc.) |
| `npc_dialogue` | Unlocks new dialogue options on an NPC |
| `zone_unlock` | Makes a zone a fast travel target |
| `vendor_unlock` | Unlocks special inventory on a vendor NPC |

## 8. LLM Generation — Prompt Strategy

### 8.1 Input Context for the LLM

When asking an LLM to generate a `MapBlueprint`, provide:

```
1. SCHEMA:   The MapBlueprint JSON schema (this document, §2)
2. GEOGRAPHY: Real-world description of the location
   - City/region name and area
   - Key landmarks (buildings, bridges, parks)
   - Water features (rivers, lakes, coast)
   - Elevation features (hills, valleys, cliffs)
   - Road layout (major streets, highways, alleys)
3. GAME CONTEXT:
   - Biome type assignment for each area
   - Difficulty tier distribution (easy near start, hard at edges)
   - Which factions control which areas
   - What quests should be available
   - Player level range for this map
4. CONSTRAINTS:
   - Chunk dimensions (e.g., 8×6)
   - Maximum building count (budget)
   - Required NPCs (with roles)
   - Required enemy group count and types
5. EXISTING ASSETS: Available prefabIds, meshAssets, dialogueIds, lootTables
```

### 8.2 Example Prompt

```
Generate a MapBlueprint JSON for the following location:

LOCATION: Downtown Portland, Oregon, USA
- The Willamette River runs north-south through the center
- Burnside Bridge and Morrison Bridge cross the river
- West side has forested hills (Washington Park area)
- East side is flat urban grid
- Pioneer Courthouse Square is a central landmark
- An old rail yard exists on the east edge

POST-APOCALYPTIC CONTEXT:
- 50 years after nuclear war
- River is polluted but not irradiated
- West hills are overgrown forest, some wildlife
- East side is ruined city, raider territory
- Pioneer Square is now "Haven Camp" — settler base
- Rail yard is a fortified raider stronghold
- A small irradiated crater where OHSU hospital was (west hills)

GAME PARAMETERS:
- Dimensions: 8 chunks wide × 6 chunks high
- Player starts at Haven Camp, level 1-3
- Difficulty: Tier 1 near Haven, Tier 3 at raider stronghold, Tier 4 at crater
- Factions: Settler's Alliance (Haven), Wasteland Raiders (east), Cult of the Glow (crater)
- 3-4 quest chains needed

AVAILABLE PREFABS: [list of prefabIds...]
AVAILABLE DIALOGUE TREES: [list of dialogueIds...]
AVAILABLE LOOT TABLES: [list of lootTables...]

Output a complete MapBlueprint JSON matching the schema.
```

### 8.3 LLM Output Validation

The generated JSON is validated before compilation:

| Check | Rule |
|-------|------|
| Schema conformance | All required fields present, types match |
| Bounds | All chunk coordinates within `chunksWide × chunksHigh` |
| Reference integrity | All `buildingRef`, `npcRef`, `enemyGroupRef`, `roadRef`, `zoneRef` exist in the blueprint |
| Asset existence | All `prefabId`, `meshAsset`, `dialogueId`, `lootTable` exist in the game's asset registry |
| Overlap | No two buildings overlap footprints |
| Walkability | Player start is on a walkable tile; all NPC/container locations are reachable |
| Quest completability | Every quest objective has matching entities in the map |

Failed validation produces a structured error report that can be fed back to the LLM for correction.

### 8.4 Iterative Refinement

```
MapBlueprint v1  →  Validator  →  errors[]
                                       │
                                       ▼
                              LLM (fix prompt + errors)
                                       │
                                       ▼
                              MapBlueprint v2  →  Validator  →  pass ✓
                                                                   │
                                                                   ▼
                                                            Map Compiler
```

Typically converges in 1–2 rounds. The fix prompt includes only the errors and the relevant sections, not the full blueprint.

## 9. Map Compiler — Blueprint to Runtime

The compiler is a deterministic, offline tool that transforms `MapBlueprint.json` into the runtime files.

### 9.1 Compilation Pipeline

```
MapBlueprint.json
   │
   ├──[1]─ Terrain Pass
   │       For each chunk (cx, cy):
   │         1. Initialize 16×16 tile grid at baseElevation
   │         2. Apply elevation regions (noise + polygon fill)
   │         3. Carve rivers (lower terrain, set WaterLevel)
   │         4. Fill lakes/craters
   │         5. Assign surface types per region rules
   │         6. Carve roads (smooth height, set surface)
   │
   ├──[2]─ Structure Pass
   │       For each building:
   │         1. Mark footprint tiles as non-walkable
   │         2. Set StructureId on footprint tiles
   │         3. Generate interior chunk if Large
   │         4. Place entrance tiles (walkable, door flag)
   │
   ├──[3]─ Entity Pass
   │       For each NPC / enemy group / container:
   │         1. Resolve placement mode to exact tile coords
   │         2. Validate tile is walkable (or interior)
   │         3. Generate EntitySpawnInfo / ContainerSpawn records
   │         4. Assign to correct chunk's entity list
   │
   ├──[4]─ Quest Wire-Up Pass
   │       For each quest hook:
   │         1. Verify referenced entities exist in entity pass output
   │         2. Generate condition flags and event wiring
   │         3. Output quest-to-entity mapping for runtime
   │
   └──[5]─ Output
           ├── world.json (dimensions, player start, tileSize)
           ├── zones.json (zone definitions)
           ├── chunks/
           │     ├── 0_0.json (16×16 tiles + entities + containers)
           │     ├── 0_1.json
           │     └── ...
           ├── chunks/interior_elder_house.json
           └── quest_hooks.json (runtime quest-entity mappings)
```

### 9.2 Output Format (per chunk)

Matches the existing `docs/schemas/world.md` format, extended for new tile data:

```jsonc
{
  "chunkX": 1,
  "chunkY": 2,
  "tiles": [
    // 16 rows × 16 columns of tile objects
    [
      { "s": 1, "h": 3, "w": 0, "st": 0, "f": 1, "v": 42 },
      // s=SurfaceType, h=HeightLevel, w=WaterLevel, st=StructureId, f=Flags, v=VariantSeed
      ...
    ]
  ],
  "entities": [
    {
      "prefabId": "enemy_raider_melee",
      "localX": 4.5,
      "localZ": 7.5,
      "rotationY": 90,
      "faction": "wasteland_raiders",
      "level": 3,
      "lootTable": "loot_raider",
      "groupId": "eg_raiders_camp",
      "tags": ["human", "hostile"]
    }
  ],
  "containers": [
    {
      "containerId": "cont_safe_office",
      "prefabId": "container_safe",
      "localX": 6.5,
      "localZ": 3.5,
      "lootTable": "loot_office_safe",
      "locked": true,
      "lockDifficulty": 3
    }
  ]
}
```

### 9.3 Compact Tile Encoding

Each tile as a full object would be verbose (256 tiles × 6 fields = large JSON). Use a compact encoding:

```jsonc
// Option A: Packed integer array (smallest)
// Each tile = single 32-bit int: [surface:4][height:8][water:8][structId:8][flags:4]
"tiles_packed": [
  [0x1030001A, 0x1030001B, ...],  // row 0
  ...
]

// Option B: Separate layer arrays (more readable, LLM-friendly)
"surface":   [[1,1,1,2,2,...], ...],   // 16×16 SurfaceType ints
"height":    [[3,3,3,3,4,...], ...],   // 16×16 HeightLevel ints
"water":     [[0,0,0,0,0,...], ...],   // 16×16 WaterLevel ints
"structure": [[0,0,0,0,0,...], ...],   // 16×16 StructureId ints
"flags":     [[1,1,1,1,1,...], ...],   // 16×16 Flags ints
"variant":   [[42,7,91,...],  ...]     // 16×16 VariantSeed bytes
```

**Recommendation:** Use **Option B** (separate layers) for human/LLM readability during development. Add Option A as an optimized binary format for shipping.

## 10. Runtime Loading

### 10.1 Load Sequence

```csharp
public sealed class MapLoader
{
    public WorldMapData LoadWorld(string mapDirectory)
    {
        // 1. Load world.json → dimensions, player start
        // 2. Load zones.json → ZoneRegistry
        // 3. WorldMapData with chunk slots (lazy)
        // Chunks loaded on demand by ChunkStreamingProcessor
    }

    public ChunkData LoadChunk(string mapDirectory, int cx, int cy)
    {
        // 1. Read chunks/{cx}_{cy}.json
        // 2. Parse tile layers → TileData[16,16]
        // 3. Parse entities → EntitySpawnInfo[]
        // 4. Parse containers → ContainerSpawn[]
        // 5. Return ChunkData
    }
}
```

### 10.2 Integration with Chunk Streaming

The existing `ChunkStreamingProcessor` publishes `ChunkLoadedEvent` when a chunk enters the 3×3 active grid. The runtime handler:

1. Calls `MapLoader.LoadChunk(cx, cy)`
2. Passes `ChunkData` to `TileMapRendererScript` for terrain/water/building mesh generation
3. Iterates `EntitySpawnInfo[]` and spawns NPC/enemy entities with appropriate components
4. Iterates `ContainerSpawn[]` and spawns interactable containers
5. Checks `ModifiedState` (save data) to skip already-looted containers or killed persistent enemies

### 10.3 Interior Chunks

When the player interacts with a door tile on a Large building:

1. `ChunkTransitionEvent` fired with `interiorChunkId`
2. `MapLoader.LoadChunk(interiorChunkId)` loads the interior chunk
3. Camera transitions to interior (roof removed / separate scene)
4. Exterior chunk remains loaded (exit door returns to exterior)

## 12. Open Questions

1. **LLM model choice:** GPT-4o / Claude for structured JSON generation? Both handle the schema well; need to benchmark accuracy on spatial reasoning (placing buildings without overlaps).
2. **Interior generation:** Should the LLM also generate interior chunk layouts, or are those hand-authored? Interior spatial design (rooms, furniture, loot placement) may be harder for an LLM to get right.
3. **Map versioning:** When the LLM regenerates a map, how do we handle save game compatibility? Player progress (modified state, looted containers) references chunk/entity IDs.
4. **Real geography source:** Plain text description vs. structured data (OpenStreetMap export, elevation GeoTIFF)? Structured data is more accurate but harder to prompt with.
5. **Compiler determinism:** The noise-based terrain generation uses seeds. Do we store the seed in the blueprint, or does the compiler choose one? Fixed seeds = reproducible maps from the same blueprint.
